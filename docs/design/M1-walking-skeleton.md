# M1 — Walking skeleton: design notes

Lightweight design record for milestone M1 (vertical slice: event ingestion → persistence,
one event type). Full SDD ceremony starts at M2; this note only captures the few real
decisions M1 introduces.

## Scope

A minimal vertical slice that proves the riskiest path (voice provider webhook → REST ingest
→ PostgreSQL) with a single event type (`call.started`). No domain modelling, no evidence
building, no scoring — those belong to M2+.

## Decision 1 — Async SQLAlchemy (asyncpg)

**Decision:** use SQLAlchemy's async engine with the `asyncpg` driver and `AsyncSession`.

**Why:** FastAPI is async and the platform's core requirement is non-blocking real-time event
ingestion (spec 5.1.4). Mixing synchronous DB calls inside async endpoints blocks the event
loop, which directly undermines ingestion throughput. Choosing async now avoids a costly
sync → async migration later.

**Trade-off:** async SQLAlchemy is more complex (async sessions, `await`, greenlet). Accepted,
because the ingestion path is the system's hot path and correctness there is the thesis.

> To be ratified before M1.3 (persistence), since it shapes all DB code.

## Decision 2 — `raw_events` table (minimal)

M1 persists the **raw** incoming event before any domain modelling. The table is intentionally
minimal and will be superseded by the full model (Event entity) in M2.

| Column        | Type        | Notes                                             |
| ------------- | ----------- | ------------------------------------------------- |
| `id`          | UUID (PK)   | Generated by the platform                         |
| `event_type`  | text        | Extracted for queryability (e.g. `call.started`)  |
| `payload`     | JSONB       | Full raw event body, unmodified                   |
| `received_at` | timestamptz | Server-side default `now()`                       |

**Why:** storing the raw body in JSONB guarantees no information is lost before the domain
model exists; `event_type` is lifted out so the trace can be queried without unpacking JSON.

## Decision 3 — Adapter / port boundary in M1

- The REST input adapter (`POST /events`) lives in `src/adapters/rest/`.
- Its input contract is the Pydantic `EventIn` schema (adapter boundary — external shape).
- Persistence lives in `src/infrastructure/` behind a thin repository.

For the skeleton the flow stays deliberately thin (adapter → repository). The full
`IngestEvent` use case and its repository **port** (interface in `application/`) are formalised
in M2.6, once the domain entities exist. M1 does not over-abstract.

## Deferred to M2

- Mapping Vapi field names → canonical governance events (e.g. `call.started` →
  `session.started`), per spec 5.1.3. In M1, `EventIn.event_type` is a free string and the
  test uses a `call.started` payload as-is.
- Domain entities (`Agent`, `Session`, `Event`), session lifecycle, and the ingest use case
  with its port.
